# Фича: story-player (новостной плеер на главной)

Единственная интерактивная фича сайта. Всё, что про плеер, лежит здесь —
если правишь плеер и открываешь файл вне этой папки, скорее всего ты не там.

## Карта папки

```
story-player/
├── StoryPlayer.svelte    ← КОРЕНЬ. Всё состояние плеера (какой пост активен,
│                            пауза, звук, режим чтения/обсуждения, режим ленты)
│                            + клавиатура, свайпы, колесо мыши. Один на страницу.
├── StoryRail.svelte      ← лента карточек слева (выбор поста)
├── StoryVisual.svelte    ← центр: фото/видео, прогресс-бар, мобильные кнопки
├── StoryContext.svelte   ← правая панель: текст поста, Read more, кнопки,
│                            вкладка обсуждения (giscus)
├── i18n.ts               ← ВСЕ строки интерфейса (en/ru). Новую надпись — сюда.
├── config.ts             ← giscus, брейкпоинт, тайминги. Константы — сюда.
├── payload.ts            ← готовит данные постов для плеера (сервер, SSR)
└── state/                → пока живёт в scripts/story-player/ (см. ниже)
```

Чистая логика (машины состояний и контроллеры) пока импортируется из
`src/scripts/story-player/` — она фреймворк-независимая и покрыта тестами:
`audio-state.ts` (звук/autoplay), `discussion-state.ts` (пауза при обсуждении),
`progress.ts` (прогресс и автопереход), `analytics.ts`, `discussion.ts` (giscus),
`media.ts` (прелоад соседних медиа). Когда старый плеер будет удалён —
перенести эти файлы в `state/` и `controllers/` сюда.

## Как это устроено (поток данных)

1. Страница (`pages/index.astro` и др.) грузит посты из БД и передаёт их в
   `components/HomeNewsPage.astro`.
2. `HomeNewsPage.astro` (Astro-слой) отвечает за SEO: noscript-статья с
   единственным `<h1>`, preload первой картинки. Плеер он рендерит как
   `<StoryPlayer client:load />` — Svelte-остров с SSR (первый кадр приходит
   готовым HTML ещё до загрузки JS).
3. Внутри острова состояние реактивное: меняешь `active` — Svelte сам
   перерисовывает rail/visual/context. НИКАКОЙ ручной работы с DOM для
   отображения состояния (никаких `element.hidden = …` — это прошлая жизнь).

## Куда класть новое

- **Новая кнопка/надпись** → строка в `i18n.ts`, разметка в соответствующий
  `.svelte`, обработчик — коллбек из `StoryPlayer.svelte`, если трогает общее
  состояние.
- **Новое состояние** (напр. «избранное») → `$state` в `StoryPlayer.svelte`;
  если логика переходов нетривиальная — чистая функция в отдельном файле
  + тест (образец: `scripts/story-player/audio-state.ts`).
- **Стили** → в `<style>` соответствующего `.svelte`-компонента (scoped).
  Все стили плеера уже там; глобальным остался только
  `styles/home-news-page.css` (body-класс и noscript-SEO — их заскоупить
  нельзя). Правила, зависящие от классов корня (`.is-reading` и т.п.), пиши
  через `:global(.story-player.is-…)`; контент, инжектируемый JS (giscus),
  тоже через `:global`. z-index — только `var(--z-*)` из
  `shared/styles/tokens.css`.
- **Совсем другая фича** (не плеер) → новая папка `features/<имя>/` с таким же
  README, страницы остаются тонкими.

## Чего здесь делать НЕЛЬЗЯ

- SEO (title, canonical, JSON-LD, h1) — это слой Astro (`HomeNewsPage.astro`,
  `layouts/Layout.astro`), плеер про SEO не знает.
- Ходить в БД — данные приходят готовыми через `payload.ts`.
- Прямые манипуляции DOM для UI-состояния — только Svelte-реактивность.
  Исключение: media API (`video.play()`), таймеры прогресса, giscus-инъекция —
  они в контроллерах.

входит в pre-push. Инструмент живёт в `tools/svelte-check/` со своим
TypeScript 5.x, потому что корень репозитория на TypeScript 7 (tsgo), чей
JS-API svelte-check пока не понимает.
