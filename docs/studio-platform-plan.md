# План развития Studio и Telegram-бота

Дата актуализации: 2026-07-14.

Этот документ описывает путь от текущего Telegram-бота к единому Studio: рабочему месту для постов, видео, очереди, аналитики и будущих AI-возможностей. Он дополняет [план надёжности](./plans.md) и [текущую архитектуру](./architecture.md).

## Цель

Нужен не «бот с большим количеством callback-ов», а одна система управления контентом:

- Telegram остаётся быстрым интерфейсом создания и контроля публикаций;
- посты и видео используют одинаковые, понятные жизненные циклы;
- очередь показывает фактическую работу: ближайшие публикации, черновики и реальные проблемы;
- аналитика отвечает отдельно за весь проект, посты и видео;
- будущие Web Studio, Command Center, AI-агенты и MCP используют те же прикладные операции;
- публикация остаётся надёжной: независимые targets, audit, защита от дублей и понятное восстановление ошибок.

## Подход: не сохранять старое любой ценой

Мы вправе переписать и существующий publishing/video/data слой, если новая модель будет проще, безопаснее и позволит быстрее развивать продукт. Ограничение другое: не делать «переписать всё одновременно».

Для каждого домена выбираем один из двух путей:

1. **Обёртка и постепенная миграция**, если текущая реализация надёжна, а проблема — в интерфейсе или связности кода.
2. **Замена ядра домена**, если текущие state-машины, таблицы или контракты мешают развитию, трудно тестируются или дублируют логику.

При замене ядра создаём новую модель рядом со старой, переносим один сценарий, сравниваем результат на fixture и production-safe read-only диагностике, затем переводим трафик. Старый путь удаляется только после периода стабильной работы и понятной миграции данных.

Проект при этом может оставаться single-process Bun-приложением с SQLite: логические сервисы и чёткие контракты важнее искусственного разнесения на микросервисы.

## Целевая схема

```text
                         studio.yaml + permissions
                                      |
Telegram bot        Web Studio        Command Center        AI / MCP
     |                   |                   |                |
     +-------------------+-------------------+----------------+
                                      |
                         Studio application services
          create · edit · preview · schedule · publish · inspect
                                      |
     +----------------+----------------+----------------+----------------+
     |                |                |                |                |
  Content          Publishing        Video        Measurement       Notifications
 drafts/posts      targets/jobs      drafts        snapshots         progress/events
     |                |                |                |                |
     +----------------+----------------+----------------+----------------+
                                      |
                       SQLite + files + platform adapters
```

Интерфейс не должен сам менять таблицы или собирать сложные запросы; он вызывает сценарий, получает типизированный результат и отображает его. Сценарий не зависит от Telegram API, HTML или конкретной кнопки.

## Принципы интерфейса

1. **Чат — для ввода, карточка — для выбора.** Текст, медиа, дата и исправления приходят обычными сообщениями. Кнопки — только для коротких развилок: режим, площадки, подтверждение, переход к разделу.
2. **Одна рабочая карточка на сценарий.** Экран редактируется, а не создаёт цепочку похожих сообщений. Новое сообщение — только результат, уведомление или запрос ввода.
3. **Контекст всегда виден.** В карточке публикации есть режим, площадки, время и статус. При редактировании видно текущее значение.
4. **Понятная ошибка возвращает на нужный шаг.** Например: «Дата не распознана — пришлите `15.07 17:00`».
5. **Меньше постоянных кнопок.** Нижняя клавиатура содержит только `☰ Menu` / `☰ Меню`; `⚙️` — короткая inline-кнопка в главном меню. Язык всегда находится в настройках, остальные пункты зависят от `studio.yaml`.
6. **Локаль — свойство пользователя.** Английский по умолчанию, русский выбирается в `⚙️`. Интерфейс не смешивает языки.
7. **Никаких ложных цифр.** Если нет baseline-снимка для периода, аналитика сообщает об этом, а не показывает одинаковые данные за 7 и 30 дней.

## Целевой UX Telegram

### Главное меню

```text
📝 Новый пост
🎬 Новое видео              ⚙️
📋 Очередь                  📊 Статистика
```

Кнопки зависят от включённых возможностей: без video-модуля не видны «Новое видео» и видео-раздел статистики. `⚙️` есть всегда и содержит язык.

### Новый пост

1. `📝 Новый пост` создаёт черновик; бот просит прислать текст с необязательным фото или видео. Есть только `Отмена`.
2. После входящего сообщения появляется карточка предпросмотра: текст, медиа, режим, число площадок и варианты RU/EN.
3. Пользователь при необходимости меняет только нужное: `Режим`, `Площадки`, `Изменить RU`, `Изменить EN`.
4. `▶️ Опубликовать сейчас` показывает короткое подтверждение с числом площадок. `📅 Запланировать` просит дату обычным сообщением, показывает распознанное время и подтверждает.
5. После запуска эта же карточка обновляется: `0 / 17 → 6 / 17 → 17 / 17`. Частичная ошибка видна по площадке; итог доступен из очереди и архива.

### Новое видео

Видео не должно требовать длинного кнопочного мастера:

1. `🎬 Новое видео` создаёт video draft и просит прислать видео.
2. Затем бот спрашивает только недостающие поля обычными сообщениями: название, описание, обложка (если нужна), время. Один текст может заполнить несколько полей, но бот показывает, что понял.
3. Рабочая карточка кратко обновляется: «Видео ✓, название ✓, описание ✓, обложка — не задана, время — сейчас».
4. В конце — компактный review с площадками и `Опубликовать` / `Запланировать`. Изменение поля просит прислать новое значение, а не открывает каскад кнопок.
5. Публикация показывает один live-progress card по video targets и сохраняет итог в архиве.

### Очередь

Очередь — рабочий inbox:

```text
📋 Очередь

Ближайшее
• Сегодня, 17:00 — Пост «…» · 17 площадок
• Завтра, 12:00 — Видео «…» · YouTube, Instagram

Черновики: 3     Требует внимания: 1
```

- По умолчанию открывается **Ближайшее**, если там есть публикации; иначе — черновики.
- Черновики — отдельный список для продолжения работы.
- «Требует внимания» появляется счётчиком только при реальной проблеме и ведёт к конкретным failed/retry items с объяснением.
- Назад, меню и фильтры редактируют ту же карточку.

### Статистика

Первый экран — общая картина, а не длинный список платформ:

```text
📊 Общая статистика · 7 дней
Подписчики: 42 380  (+1 240)
Просмотры: 94 200  (+18%)
Взаимодействия: 4 239  (+11%)

Сегодня · 7 дней · 30 дней
📝 Постинг       🎬 Видеопостинг
← Меню
```

- `📝 Постинг`: текстовые публикации, топ-посты, `Архив постов`.
- `🎬 Видеопостинг`: video-площадки, топ-ролики, `Архив роликов`, при наличии — анализ аудитории.
- Детали платформ раскрываются по запросу, не шумят на первом экране.
- Вместо технического API-текста: «Обновлено сегодня, 09:30», только когда свежесть данных важна.

## Этапы перехода

Каждый этап выпускается отдельно, за feature flag/внутренней ролью, с regression-тестами и возможностью вернуть прежний путь.

| Этап | Пользовательский результат | Технический результат | Критерий готовности |
| --- | --- | --- | --- |
| 0. Стабилизация UX | Единые язык, меню, навигация, честная статистика | i18n, builders экранов, callback-тесты | Нет смешения локалей; некорректные дельты не показываются |
| 1. Контракты сценариев | Одинаковое поведение поста во всех экранах | Services для draft, schedule, publish, queue | Bot не содержит сценарную DB-логику |
| 2. Studio shell | Карточки редактируются стабильно | `Screen`, session state, callback codec, renderer | Один активный экран; устаревшая кнопка безопасна |
| 3. Posts v2 | Ввод сообщениями, preview, confirm, progress | Новый post-flow поверх старого или нового publishing core | Create/edit/schedule/partial failure покрыты E2E |
| 4. Analytics v2 | Общая / Постинг / Видеопостинг, честные дельты | Metric service, coverage, snapshot policy | У периода есть baseline или ясное сообщение |
| 5. Video v2 | Быстрый диалоговый flow и resume draft | Video draft service, field-level commands | Нет обязательного длинного button wizard |
| 6. Queue и уведомления v2 | Ближайшее, черновики, проблемы, live-progress | Unified activity/event read-model | Один предмет имеет единый статус в интерфейсах |
| 7. Замена старого publishing/video ядра, если оправдано | Надёжнее и проще развивать delivery | Новые state machines/repositories, миграция и dual-read/compare | Нет регрессий, migrate/rollback доказаны |
| 8. Web Studio | Те же действия в браузере | Web использует те же services | Telegram и web дают одинаковый audit trail |
| 9. AI и MCP | AI помогает, но не обходит контроль | Typed tools, permissions, confirmations/audit | AI проходит те же preview и publish services |

## Ближайшие фичи, которые можно делать уже сейчас

1. Вынести все видимые строки и подписи кнопок в RU/EN каталог; тестировать рендер каждого экрана в обеих локалях.
2. Свести главное меню, настройки и `← Меню` к одному renderer-у: постоянная клавиатура — только `☰ Menu` / `☰ Меню`, настройки — `⚙️` inline.
3. Оставить `Language` всегда; YouTube signature и другие настройки показывать только при включённом модуле из `studio.yaml`.
4. Перестроить очередь: default `Ближайшее`, отдельные черновики, счётчик реальных проблем, редактирование одной карточки вместо удаления и новой отправки.
5. Исправить периодную аналитику: baseline для 1/7/30 дней, состояние «история ещё собирается», production-совместимая fixture для запросов.
6. Сделать Analytics v2 UI: короткий общий экран, разделы «Постинг» / «Видеопостинг», свой архив внутри каждого, без технического API-текста.
7. Вынести post/video archives в самостоятельные read-model/use-case функции с пагинацией и локализованными статусами.
8. Ввести минимальные Studio contracts: `CreatePostDraft`, `UpdatePostDraft`, `SchedulePublication`, `StartPublication`, `GetQueue`, `GetAnalytics`. Сначала они могут вызывать нынешний код.
9. Ввести screen/session слой для новых экранов без переноса legacy за один релиз.
10. Добавить Telegram E2E: меню, язык, back-навигация, post create/edit/schedule, video resume, очередь, прогресс 17 targets и partial failure.

## Первые поставки

### Релиз A — текущий бот без несостыковок

- Полная локализация analytics, archives, audience analysis, ошибок и кнопок.
- Единый renderer меню/настроек/возврата.
- Корректная причина одинаковых 7/30; честный fallback при отсутствии истории.
- Короткая общая статистика и разделение на Постинг / Видеопостинг.

### Релиз B — очередь и навигация

- Queue read-model: upcoming, drafts, attention только при необходимости.
- Редактирование control cards и политика, когда создаётся новое сообщение.
- Унифицированные архивы постов и видео.

### Релиз C — Posts v2

- Application service для draft/edit/schedule/publish.
- Dialog-first flow с preview, confirm и одной live-progress карточкой.
- Старый flow — fallback до пользовательской проверки нового.

### Релиз D — Video v2

- Диалоговый сбор полей, field-level edit, resume draft.
- Общий с постами механизм review, confirmation и progress.

## Технические контракты

### Application services

Команда принимает actor, entity/ref и валидированные данные; возвращает доменный результат, не Telegram markup.

```ts
type SchedulePublication = (input: {
  actorId: string;
  ref: PostRef | VideoDraftRef;
  when: Date;
}) => Promise<{ scheduleId: string; scheduledFor: Date }>;
```

Ошибки прикладные (`invalid_schedule_time`, `draft_not_ready`, `not_allowed`), чтобы RU и EN объясняли одну причину одинаково.

### Read models

Запросы возвращают готовое для экрана состояние: `QueueItem`, `PostSummary`, `VideoSummary`, `AnalyticsOverview`, `AnalyticsCoverage`. Они не меняют данные и подходят Telegram, web и AI.

### События и уведомления

События фиксируют факты: `publication.started`, `target.succeeded`, `target.failed`, `publication.completed`, `metrics.refreshed`. Notification policy решает, обновить ли control card, отправить отдельное уведомление или ничего не делать. Поэтому публикация на 17 площадок имеет одно обновляемое сообщение.

### Права и AI

У действия есть actor, источник (`telegram`, `web`, `mcp`) и audit record. AI/MCP может читать, готовить черновик и предлагать план; запуск публикации проходит те же права и подтверждение, что Telegram.

## Когда оправдано переписать ядро

Переписываем publishing/video domain не ради папок, а если выполнено хотя бы одно условие:

- state transitions трудно понять или проверить без интеграции с Telegram/платформой;
- одинаковые правила schedule/retry/target status реализованы в нескольких местах;
- данные не позволяют ответить «что произошло с публикацией» одним запросом;
- добавление новой площадки или типа контента требует менять несвязанные слои;
- невозможно надёжно обеспечить idempotency, audit и migration/rollback.

План замены: сначала формализовать state machine и invariants; затем новую схему и repositories; импорт/dual-read; shadow compare; rollout одного типа публикации; только после подтверждённого recovery и rollback — перенос остальных и удаление legacy.

## Модель данных и миграции

- Authoritative tables не переименовываются ради красоты; изменения — append-only SQL migrations.
- Session state Telegram короткоживущее, имеет TTL и версию; устаревший callback предлагает открыть меню, а не падает.
- Screens и callbacks используют стабильные короткие IDs, а не сериализованный пользовательский текст.
- Analytics хранит время snapshot и coverage по источнику; delta вычисляется только при достаточной истории.
- Web/AI не получают secrets, platform tokens, внутренние error payloads или произвольный SQL.

## Качество и rollout

Перед включением новой фичи:

1. Unit-тесты use-case и локализации.
2. Contract-тесты для state transitions и platform publishers.
3. Telegram E2E-smoke: callback-навигация, повторный update, устаревшая кнопка.
4. Feature flag сначала для owner/admin, затем для всех.
5. Наблюдение через audit, `ops doctor`, `ops audit`, `ops verify` и очередь. Legacy удаляется только после стабильного периода.

## Открытые продуктовые решения

- Какие общие метрики между платформами можно честно суммировать, а какие нет (например, уникальная аудитория).
- Часовой пояс по умолчанию и точность времени для планирования.
- Какие действия AI может выполнять без подтверждения, а какие всегда требуют preview/confirm.
- Web Studio сначала только для просмотра/черновиков или сразу для запуска публикаций.
- Retention для sessions, control cards, notification events и analytics snapshots.

## Definition of done для Studio-фичи

Фича готова, когда она:

- работает в RU и EN без смешения строк;
- учитывает включённые/выключенные модули `studio.yaml`;
- имеет состояния загрузки, пустого списка и ошибки;
- не создаёт дубль публикации при повторном callback/update;
- сохраняет audit и не обходит publish jobs/publishers;
- покрыта пользовательским тестом и проверена production-safe диагностикой после деплоя.
